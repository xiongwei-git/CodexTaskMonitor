# AgentTaskMonitor 初步可行性评估

评估日期：2026-07-20

## 结论

**建议推进，决策标记为 `GO`。**

这是一个由真实高频工作流产生的产品问题，不是为了使用新技术而寻找场景。用户同时维护约 20 个项目，并行运行多个 Codex 任务；现有的每日项目梳理只能生成某个时间点的快照，无法回答以下即时问题：

- 哪些任务现在仍在工作？
- 哪些任务已经正常完成？
- 哪些任务正在等待用户输入或审批？
- 哪些任务可能因 Codex 或应用异常而中断？
- 该回到哪个对话继续处理？

“项目多、任务多、执行时间长、运行状态不可见”共同形成了明确痛点。第一版不需要代替 Codex App，只需要成为它的本地实时总览和异常入口。

## 可行性评分

| 维度 | 判断 | 说明 |
|---|---|---|
| 痛点真实性 | 高 | 用户已经在并行使用中持续遇到查找和状态不透明问题 |
| 用户价值 | 高 | 能直接减少反复查找、无效等待和异常漏看 |
| 项目清单可获取性 | 高 | 本机 Codex 状态数据库包含线程、工作目录、标题和更新时间 |
| 点击回到 Codex | 高 | 官方支持通过 `codex://threads/<thread-id>` 打开本地对话 |
| 现有 App 实时状态准确性 | 中 | 外部进程无法直接共享当前桌面 App 进程内的运行态，需要观察事件文件并进行有限推断 |
| 异常终止判断 | 中 | 可识别缺失终止事件和长时间无活动，但必须保留“疑似异常”状态，避免误报 |
| 产品扩展空间 | 高 | 后续可增加通知、等待处理、耗时统计、跨 Agent 支持和任务控制 |

## 本机验证结果

本次评估直接检查了当前机器上的 Codex 本地状态，没有只依赖产品设想。

### 1. 任务与项目元数据可以读取

本机 `~/.codex/state_5.sqlite` 当前包含：

- 145 条 Codex 线程记录
- 70 个不同工作目录
- 126 条未归档线程
- 其中 `~/CodeX` 下有 97 条线程，分布在 46 个工作目录，82 条未归档

这说明用户感受到的“约 20 个项目”在底层已经扩张成更多工作目录和线程记录。看板不能简单地把每条线程当成一个项目，必须先完成项目归一化。

线程表能够提供：

- 技术线程 ID
- 工作目录
- 标题与预览
- 创建和更新时间
- 是否归档
- Git 信息
- 来源和模型信息

但该表本身没有持久化的“正在运行”字段。

### 2. Codex 协议原生具备理想状态模型

本机 Codex 版本为 `codex-cli 0.145.0-alpha.18`。根据该版本生成的 app-server 协议定义：

- 线程状态包括 `notLoaded`、`idle`、`active`、`systemError`
- `active` 状态可附带 `waitingOnApproval`、`waitingOnUserInput`
- Turn 状态包括 `completed`、`interrupted`、`failed`、`inProgress`
- 协议会发送 `thread/status/changed`、`turn/started`、`turn/completed` 等实时通知

如果任务由 AgentTaskMonitor 自己连接的 app-server 启动，这些状态可以被准确、实时地消费。

### 3. 不能直接从另一个 app-server 读取桌面 App 的实时态

当前 ChatGPT/Codex 桌面 App 启动了一个独立的 Codex app-server，并通过私有的标准输入输出管道连接。

本次另起一个只读 app-server，读取最近 100 条线程时：

- 100 条全部返回 `notLoaded`
- 当前正在执行本次评估的线程也返回 `notLoaded`

因此，**仅调用另一个 app-server 的 `thread/list`，无法得知现有桌面 App 中哪些线程正在工作。** 这是第一版最重要的技术边界。

### 4. 本地会话事件可以支撑观察型 MVP

`~/.codex/sessions/**/*.jsonl` 会持续写入任务事件，包括：

- `task_started`
- `task_complete`
- `turn_aborted`
- 工具调用结束、消息、推理和 token 更新等活动信号

当前正在进行的评估线程已出现 `task_started`，尚未出现后续 `task_complete`，并且文件仍在增长，因此可以识别为正在工作。

对现有会话文件做只读扫描时，发现：

- 大部分会话以 `task_complete` 结束
- 当前会话处于最近活动但未结束状态
- 另有 3 个历史会话留下了较旧的未闭合 `task_started`
- 其中 1 个有明确 `turn_aborted: interrupted`，另外 2 个缺少明确终止信号

这恰好说明产品价值，也说明异常判断不能只看一个字段。应把“缺少完成事件 + 长时间无文件活动 + 对应运行进程不可见”等信号组合起来，标记为“疑似异常终止”。

## 产品对象必须分成三层

用户口中的“项目”和 Codex 底层的“任务”不是一回事。建议明确三个对象：

1. **项目 Project**
   - 例如 `~/CodeX/AgentTaskMonitor`
   - 是用户管理工作的长期容器
2. **任务 Thread**
   - Codex 中的一条对话
   - 一个项目可以有多条任务
3. **执行 Turn**
   - 用户在任务中发出一次请求后，Codex 的一次实际运行
   - 实时状态真正属于这一层

看板默认按项目聚合，再展开查看任务和最近一次执行。否则 100 多条历史线程会再次造成信息过载。

## 推荐状态模型

不建议只使用“工作中 / 停止”两个状态。“停止”会混淆正常完成、等待用户和异常终止。

| 看板状态 | 含义 | 建议颜色 |
|---|---|---|
| 正在工作 | 最近一次 Turn 已开始，事件仍在持续 | 绿色 |
| 等待处理 | 等待用户输入、审批或授权 | 橙色 |
| 已完成 | 收到正常完成事件 | 蓝色 |
| 已中断 / 失败 | 收到明确的中断或失败事件 | 红色 |
| 疑似异常 | 已开始但长时间无活动，也没有正常终止事件 | 红色警示 |
| 空闲 | 当前没有运行中的 Turn | 灰色 |
| 状态未知 | 数据不足或 Codex 版本不兼容 | 浅灰色 |

项目卡片的状态由其任务聚合而来，优先级建议为：

`疑似异常 > 等待处理 > 正在工作 > 已中断/失败 > 已完成 > 空闲`

## 推荐 MVP

第一版定位为：**本地只读的 Codex 项目与任务实时看板。**

### 必须具备

1. 自动发现 `~/CodeX` 下的项目及关联 Codex 线程。
2. 按项目显示正在工作、等待处理、疑似异常、空闲等聚合状态。
3. 展开项目后显示最近任务、最近活动时间和运行时长。
4. 通过 `codex://threads/<thread-id>` 一键打开对应 Codex 对话。
5. 支持“全部、正在工作、需要处理、疑似异常”筛选。
6. 只读取元数据和事件类型，默认不读取或展示对话正文。

### 暂不进入第一版

- 从看板直接暂停、恢复或发送消息
- 云端同步
- 团队协作
- 跨设备监控
- 对 Codex 之外其他 Agent 的兼容
- 复杂的成本和生产力分析

## 推荐技术路线

### 路线 A：本地观察器

**第一版推荐。**

读取以下只读数据：

- `state_5.sqlite`：项目与线程清单
- `sessions/**/*.jsonl`：开始、完成、中断和活动事件
- Codex 进程状态：辅助判断应用是否仍在运行

优点：

- 不改变用户现有 Codex App 使用方式
- 可以覆盖已经存在的项目和对话
- 适合快速验证看板价值

限制：

- 依赖本地内部文件格式，Codex 升级后可能需要适配
- “异常终止”是多信号推断，不能承诺绝对准确

### 路线 B：由 AgentTaskMonitor 托管 Codex 任务

通过 Codex app-server 启动和管理任务，直接消费官方实时通知。

优点：

- 状态最准确
- 后续可以支持暂停、继续、输入和审批

限制：

- 用户需要从 AgentTaskMonitor 发起任务，改变现有工作习惯
- 产品复杂度和安全责任明显增加

### 推荐演进

先做路线 A 验证需求，再逐步加入路线 B。最终可以形成混合模式：

- 现有 Codex App 任务：只读观察
- 从 AgentTaskMonitor 启动的任务：精确监控和控制

## 风险与控制

### 1. Codex 内部格式变化

风险：本地数据库和 JSONL 结构可能随版本变化。

控制：

- 建立独立数据适配层
- 启动时识别 Codex 版本
- 无法识别时显示“状态未知”，不要错误判断
- 保留样本测试，随 Codex 升级运行兼容性检查

### 2. 异常误报

风险：长时间工具调用、等待审批或电脑休眠可能被误判为异常。

控制：

- 结合事件类型、文件活动、等待标记、进程状态和电脑休眠状态
- 使用“疑似异常”，不直接声称“Codex 已崩溃”
- 允许用户调整超时阈值

### 3. 项目归类错误

风险：临时目录、worktree、ChatGPT Project 镜像和长期 `~/CodeX` 项目可能属于同一业务项目。

控制：

- 以 `~/CodeX` 项目目录为主目录
- 结合 Git remote、工作目录和用户维护的项目注册信息归一化
- 允许手工合并、忽略和重命名

### 4. 隐私

风险：会话文件可能包含提示词、代码、命令输出或敏感信息。

控制：

- 第一版仅解析事件类型、时间和技术 ID
- 默认不上传任何数据
- 不复制或索引对话正文
- 所有读取操作保持只读

## 成功标准

MVP 值得继续推进，需要证明：

1. 用户打开看板后，10 秒内能判断哪些项目需要关注。
2. 正常运行和完成状态在数秒内更新。
3. 能发现至少一类现有 Codex App 不容易察觉的中断或停滞。
4. 点击任务能可靠回到正确 Codex 对话。
5. 连续使用一周后，用户减少在 Codex App 中反复搜索任务的行为。

## 最终建议

这个创意的正确切入点不是“管理所有 Agent”，而是：

> **先为高频并行使用 Codex 的个人用户，提供一个本地、实时、可回到原对话的项目任务雷达。**

产品价值顺序应当是：

1. 找得到
2. 看得懂
3. 知道哪里需要处理
4. 能回到原任务
5. 最后才是从看板直接控制 Agent

以这个顺序推进，第一版范围可控，且每一步都能直接验证用户价值。
