# AgentTaskMonitor 产品发现记录

更新时间：2026-07-22

## 1. 创意原点

用户目前使用 Codex 同时维护约 20 个项目，并经常并行推进多个任务。

随着项目和任务数量增长，Codex App 中查找具体任务越来越困难。用户已经设置每日定时任务梳理项目，但日报只能提供定时快照，仍无法知道：

- 每个项目的最新实时状态
- 哪个 Codex 任务正在工作
- 哪个任务已经停止
- 哪个任务可能在执行中异常终止
- 如何快速回到对应任务对话

由此产生了可视化实时看板的想法。

## 2. 问题陈述

> 当重度 Codex 用户同时维护多个项目并并行运行多个任务时，因为 Codex App 缺少跨项目实时总览和异常感知，用户难以判断任务正在运行、已经完成、等待处理还是异常中断，导致反复查找、无效等待和任务遗漏。每日项目梳理无法解决该问题，因为它只能提供定时快照。

## 3. 目标用户

首位目标用户：

- 同时维护十多个本地项目
- 经常并行运行多个 Codex 任务
- 任务可能持续较长时间
- 需要快速发现中断或等待处理状态
- 当前主要通过 Codex App 管理任务

第一阶段应先服务个人重度用户，不急于扩展到团队或平台方。

## 4. 用户任务与痛点

| 场景 | 用户想完成什么 | 当前做法 | 具体痛点 | 严重程度 | 证据状态 |
|---|---|---|---|---|---|
| 多项目并行 | 快速知道当前哪些项目正在推进 | 在 Codex App 中逐项查找 | 项目和任务多后难以定位 | 高 | 用户确认 |
| 长任务执行 | 判断 Codex 是否仍在有效工作 | 打开对话查看 | 无统一实时状态 | 高 | 用户确认 |
| 异常处理 | 及时发现任务异常终止 | 等用户偶然注意到 | 中断可能长期不被发现 | 高 | 用户确认 |
| 项目复盘 | 了解每个项目最新状态 | 每日定时梳理 | 只有快照，不是实时状态 | 中 | 用户确认 |
| 回到任务 | 从总览快速继续某个任务 | 在 Codex App 中搜索 | 上下文切换成本高 | 高 | 用户确认 |

## 5. 初步机会方向

以下只作为讨论导航，不代表产品范围：

- 看见：统一展示 Agent 正在做什么、做到哪一步。
- 理解：把日志和工具调用转化为人能快速判断的状态。
- 提醒：在完成、失败、停滞或需要输入时通知用户。
- 介入：允许暂停、继续、调整优先级或补充上下文。
- 复盘：统计耗时、失败模式、成本和结果质量。

## 6. 关键产品假设

| 假设 | 为什么重要 | 如何验证 | 当前状态 |
|---|---|---|---|
| 用户无法可靠判断 Agent 是否仍在有效工作 | 决定监控是否存在刚需 | 用户当前工作流与本机数据检查 | 已初步确认 |
| 每日项目快照不能替代实时状态 | 决定产品是否区别于现有自动化 | 对比两种使用场景 | 已确认 |
| 项目、任务和单次执行必须分层 | 决定看板信息架构 | 对照本机线程与工作目录 | 已确认 |
| 本地事件能够支持可用的状态观察 | 决定 MVP 技术可行性 | 真实 Codex 数据库、会话事件与状态检测器原型联合验证 | 已验证（个人环境） |
| 用户愿意为关键状态主动接收提醒 | 决定通知是否是核心能力 | 原型测试 | 待验证 |

## 7. 产品边界

待确认：

- 首批支持哪些 Agent 或平台？
- 只读监控还是允许控制？
- 本地优先、云端服务，还是混合模式？
- 是否处理敏感的任务上下文和日志？
- 单用户工具还是团队协作产品？

## 8. MVP 定义

初步 MVP：本地只读的 Codex 项目与任务实时看板。

MVP 应回答：

> 用户是否因为更早发现任务状态或异常，而减少等待、误判或重复操作？

建议只验证：

- 自动发现项目与任务
- 正在工作、正常完成、疑似异常三类核心检测
- 按项目聚合的看板
- 一键回到 Codex 对话

## 9. 讨论日志

### 2026-07-20

- 创建项目空间。
- 用户描述了多项目并行、任务难以查找、实时状态不可见和异常中断不易察觉的问题。
- 初步定义为本地 Codex 项目与任务实时看板。
- 完成对本机 Codex 数据库、事件文件、app-server 状态协议和深链接的验证。
- 初步可行性判断为 `GO`，等待用户决定是否进入原型阶段。

### 2026-07-21

- 推荐产品形态为 macOS 本地桌面伴侣应用。
- 用户主要通过菜单栏感知全局状态，通过桌面看板查看项目与任务详情。
- 产品内部可以使用 Web UI；localhost 页面只作为原型，不作为最终交付。
- 第一版不采用云端 Web 服务、浏览器扩展、Codex 插件或完整 Agent 客户端形态。
- 用户提出本地运行服务、通过 API 提供数据、使用 Web 页面呈现。
- 初步判断该方案适合作为核心内部架构：REST 提供快照，SSE 推送实时状态，前端不直接读取 Codex 内部文件。
- 完成本地看板原型：状态总览、项目卡片、筛选搜索、关注动态和 Codex 深链接均已接入真实数据。
- 当前 ChatGPT Project 已通过声明名称精确归入长期 `AgentTaskMonitor` 项目；不进行模糊名称猜测。
- 看板选择状态优先的“本地任务指挥台”，暂不加入趋势图、任务控制或复杂多页面导航。
- FeishuKnowNexus 暴露出新的关键场景：Codex 对话已完成，但由它启动的后台迁移仍在运行，原看板因此把任务状态误当成项目状态。
- 产品模型调整为三层：项目聚合状态、Codex 任务状态、项目后台进程状态。第一版用项目路径匹配长时进程，并对完整命令和参数做严格脱敏。
- 后台进程存在时项目显示“工作中”；短时间内重复替换进程时显示“频繁重启／疑似异常”。业务进度暂不通过任意日志扫描推断。
- 用户反馈原页面交互不够简洁直观、卡片过大，并要求加入可视化元素。
- 看板据此改为桌面三列紧凑卡片；新增可点击的项目状态分布条、卡片任务状态轨道和后台运行波形，并在无异常时自动收起关注区域。
- 用户希望每个项目有一句话概要，以便不打开详情就能理解项目用途。
- 简介由 ProjectNavigator 生成并标记来源质量，看板卡片固定显示两行；资料不足时明确待补充，不根据名称猜测。
- 用户确认开发 Web 端项目优先级排序。采用 P0–P3 人工优先级、同级手工顺序和独立本机偏好文件；异常关注与人工优先级保持分离。

### 2026-07-22

- 使用真实 Codex 状态再次验证本地观察链路：探针读取 86 个未归档线程和 86 个会话文件，无文件缺失或解析错误，并能区分当前工作中与已完成任务。
- 未关联任务已完成治理，真实探针结果由原记录的 43 个降为 0；“本地事件能够支持可用状态观察”在当前个人环境中由待验证更新为已验证。
- 验证结论目前只覆盖首位目标用户的本机环境；跨用户、跨版本稳定性仍需后续验证。
