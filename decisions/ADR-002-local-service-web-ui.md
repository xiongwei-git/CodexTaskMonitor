# ADR-002: 使用本地服务 API 与 Web 前端分离架构

## Status

Accepted

## Date

2026-07-21

## Context

AgentTaskMonitor 需要同时完成两类差异明显的工作：

1. 持续读取 Codex 本地数据库、会话事件和进程状态，并将多个底层信号转换为稳定的产品状态。
2. 通过可视化页面按项目展示状态、筛选需要处理的任务，并跳转到 Codex 对话。

如果 Web 页面直接读取或理解 Codex 的内部文件格式，前端会与 Codex 版本和本机路径强耦合，状态逻辑也难以测试和复用。

用户提出让本地服务通过接口提供数据，再由 Web 页面呈现。这与本项目的本地优先和只读观察边界一致。

## Decision

采用以下内部架构：

> **本地观察服务是唯一状态引擎和产品数据源，Web 前端只通过版本化 API 获取归一化数据。**

### 组件边界

#### 本地观察服务

负责：

- 只读访问 Codex 状态数据库和会话文件。
- 监听文件变化并维护任务状态机。
- 结合进程状态判断活动、完成、中断和疑似异常。
- 将工作目录、worktree 和临时目录归一化为项目。
- 生成稳定的 Project、Thread、Turn 产品对象。
- 提供 REST 快照接口和 SSE 实时事件流。
- 仅写入 AgentTaskMonitor 自己的配置、映射和状态历史，不修改 Codex 数据。

#### Web 前端

负责：

- 项目总览和任务详情。
- 状态筛选、搜索和排序。
- 显示状态依据和数据质量。
- 通过 API 返回的深链接打开 Codex 对话。
- 不直接读取 Codex SQLite、JSONL 或系统进程。

### 通信方式

- REST API：页面首次加载、刷新、搜索和详情查询。
- Server-Sent Events：服务向页面单向推送状态变化。
- 第一版不使用 WebSocket，因为产品暂时不需要页面向后台持续发送控制指令。

### 部署形态

#### 原型阶段

- 本地服务同时提供静态 Web 页面和 `/api/v1` 接口。
- 用户通过 `http://127.0.0.1:<port>` 访问。
- 前端和 API 保持同源，不开放跨域访问。

#### 正式桌面版本

- 保留同一套本地服务和 API 合同。
- Web 页面嵌入桌面应用窗口。
- 菜单栏、自动启动和系统通知由桌面壳提供。
- 这使原型进入桌面版时不需要重写状态引擎和页面。

## API Boundary

API 只提供 AgentTaskMonitor 的产品数据，不提供“读取任意文件”或“返回原始会话日志”的通用能力。

初步资源：

- `/api/v1/overview`
- `/api/v1/projects`
- `/api/v1/projects/{projectId}`
- `/api/v1/threads`
- `/api/v1/threads/{threadId}`
- `/api/v1/sources`
- `/api/v1/events`
- `/healthz`

详细合同记录在 `docs/api-contract-v1.md`。

## Security Boundary

第一版虽然只在本机运行，但本地服务仍视浏览器和请求输入为不可信。

必须满足：

- 仅监听 `127.0.0.1` 或 `::1`，绝不默认监听 `0.0.0.0`。
- 前端和 API 同源，不返回通配符 CORS。
- 校验 `Host` 和 `Origin`，拒绝来自任意网站的跨源调用。
- 不在 URL、日志或前端存储中暴露访问令牌。
- 不返回提示词、消息正文、命令输出或原始日志。
- 所有列表接口限制最大页大小。
- 所有路径、筛选和排序参数在服务边界校验。
- SQLite 查询参数化，不拼接外部输入。
- 错误响应不返回本机路径、堆栈或内部数据库结构。

如果以后增加任务控制、远程访问或局域网访问，必须单独设计身份认证、授权和防重放机制，不能沿用只读 localhost 的安全假设。

## Alternatives Considered

### 1. Web 页面直接读取本地文件

不采用。

原因：浏览器权限不足，且会让页面依赖 Codex 内部格式，无法形成稳定状态模型。

### 2. 前端直接查询 Codex SQLite

不采用。

原因：数据库没有完整的实时运行状态，前端还需要理解会话事件和进程状态；这会把业务逻辑分散在界面层。

### 3. 页面定时轮询全部数据

不作为主要实时机制。

原因：实现简单但更新延迟和重复读取较多。保留低频轮询作为 SSE 断线后的恢复手段。

### 4. 第一版直接使用 WebSocket

不采用。

原因：当前只需要服务端向页面单向推送状态。SSE 更简单，浏览器原生支持自动重连，足以满足需求。

### 5. 云端 API

第一版不采用。

原因：数据源在本机，且当前没有跨设备和团队协作需求。

## Consequences

### Positive

- 状态采集和界面开发可以独立演进。
- Codex 内部格式变化被封装在服务适配层。
- localhost 原型可以平滑进入桌面应用。
- 同一 API 将来可以服务菜单栏、桌面看板和可选远程客户端。
- 服务层可以进行状态机和兼容性自动测试。

### Negative

- 引入本地服务生命周期、端口和安全边界管理。
- 需要维护 API 版本和前后端兼容性。
- SSE 断线重连和事件补发需要明确设计。
- 正式桌面版需要协调后台服务与桌面壳的启动和退出。

## Revisit When

- 第一版需要从页面直接控制 Codex 任务。
- 需要局域网、手机或跨设备访问。
- 需要团队共享和用户权限。
- Codex 提供稳定的统一状态订阅接口。
- SSE 无法满足事件频率或双向交互需求。
