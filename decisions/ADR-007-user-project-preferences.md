# ADR-007: 将人工项目优先级保存为独立本机偏好

## Status

Accepted

## Date

2026-07-21

## Context

用户需要直接在看板中设置项目优先级并调整同级顺序。现有 `project-registry.json` 由 ProjectNavigator 自动生成，不适合保存人工操作；浏览器 localStorage 又无法跨浏览器复用，也不便备份和审计。与此同时，Codex 数据读取必须继续保持只读。

## Decision

1. 在 ProjectNavigator 中新增用户维护文件 `project-preferences.json`，与自动生成的登记表分离。
2. 项目键使用相对于 `~/CodeX` 的一级目录名；文件仅保存 `priority`、`rank` 和 `updated_at`。
3. 优先级固定为 `p0`、`p1`、`p2`、`p3`；`unset` 表示删除该项目偏好。
4. 同级顺序使用间隔为 1000 的整数 rank；重排时统一重新编号。
5. AgentTaskMonitor 对 Codex 状态继续只读，只允许通过固定 API 修改这一份偏好文件。
6. 写请求必须来自 loopback 同源页面，使用 JSON，限制请求体大小，并按当前快照验证项目 ID。
7. 文件通过同目录临时文件原子替换，权限为 `0600`；每次覆盖前将上一版保存为 `.bak`。
8. 异常状态提醒与人工优先级分开：关注区域始终显示异常，项目网格默认按人工优先级排序。
9. 页面在优先级选择和保存期间延迟 SSE、定时刷新；与交互重叠的在途刷新结果不得重建项目卡片。

## Alternatives Considered

- 写入 `project-registry.json`：自动扫描会覆盖人工设置，职责混乱，因此拒绝。
- 使用浏览器 localStorage：无法跨浏览器共享，丢失和迁移风险高，因此拒绝。
- 立即使用 SQLite：当前只有几十个项目和少量字段，增加迁移与运维成本，暂不采用。

## Consequences

- 人工设置不会被每日 ProjectNavigator 扫描覆盖，并且可以直接备份或手工检查。
- 本地服务从完全只读变为“Codex 数据只读、项目偏好窄范围可写”。
- 项目改名后需要迁移对应项目键；未来如加入标签、备注和历史记录，可再迁移到 SQLite。
- 页面支持原生优先级选择、同级拖拽和键盘可用的上移／下移按钮。
- 自动刷新不会关闭正在操作的优先级选择器；操作取消后补一次刷新，保存成功后等待下一份实时快照同步。
