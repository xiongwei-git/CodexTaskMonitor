# ADR-005: 项目状态与 Codex 任务状态分离，并观察后台进程

## Status

Accepted

## Date

2026-07-21

## Context

FeishuKnowNexus 的 Codex 对话已经产生 `task_complete`，但该对话启动的迁移脚本仍作为脱离 Codex 的后台进程运行。因此，仅用 Codex 会话事件表示项目状态，会把“任务已完成、项目仍在执行”错误显示成整个项目已完成。

后台进程的完整命令行可能包含访问令牌、文件参数或其他敏感信息，不能直接写入日志或 API。

## Decision

1. 分别保存项目聚合状态、Codex 任务状态和项目后台运行状态。
2. 只观察命令路径明确位于 `~/CodeX/<project>` 的长时进程，并归入该一级项目目录。
3. 进程至少持续 10 秒才进入看板，以过滤短命令和构建噪声。
4. API 只返回不透明 ID、运行时类型、脚本文件名、开始时间、推断状态和重启次数；不返回 PID、完整命令或参数。
5. 后台进程存在时，项目聚合状态为“工作中”，但已经完成的 Codex 任务仍保持“已完成”。
6. 同一进程身份在 5 分钟内发生两次 PID 替换时，标记为“频繁重启／疑似异常”。重启历史只保存在服务内存中。
7. 第一版不读取任意项目日志来推断业务成功或失败。

## Consequences

- 可以准确表达“Codex 已交棒给后台程序”的真实状态。
- 看板不会因展示进程而泄露命令参数和令牌。
- 服务重启后需要重新积累重启历史；首次观察只能显示“后台运行”。
- 由 shell 包装、路径不在项目目录或运行不足 10 秒的进程可能不会被识别。
- 后续如果要识别业务进度，应采用项目显式上报协议，而不是扫描任意日志正文。
