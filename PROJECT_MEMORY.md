# PROJECT_MEMORY: AgentTaskMonitor

## 项目概述

AgentTaskMonitor 是一个处于创意发现阶段的长期产品项目。

项目空间由用户于 2026-07-20 发起，目的是与 Codex 持续探讨并沉淀一个产品创意。

当前产品方向已初步明确：为同时维护多个项目、并行运行多个 Codex 任务的高频用户，提供一个本地可视化看板，汇总项目和任务的实时状态、提示异常，并可一键返回对应 Codex 对话。

## 已确认事实

- 长期项目目录：`~/CodeX/AgentTaskMonitor`
- 项目名称：`AgentTaskMonitor`
- 当前阶段：技术原型
- 首位目标用户就是项目发起者本人。
- 用户当前约有 20 个 Codex 项目，并经常并行推进多个任务。
- Codex App 在项目和任务增多后查找困难。
- 每日定时项目梳理只能提供快照，无法呈现实时运行状态。
- 用户无法及时知道某个 Codex 任务是否正在工作、已经停止或异常终止。
- 用户希望点击看板中的任务后进入对应 Codex 对话。
- 用户希望先进行评估，而不是立即进入开发。
- 用户反馈原看板卡片过大、交互不够简洁直观，并希望增加可视化元素。
- 用户要求项目卡片尺寸保持一致，不能因后台进程或任务内容多少产生高度差。
- 用户希望每个项目有一句话概要，让人不依赖项目名也能快速理解用途。
- 用户希望直接在 Web 看板中设置项目优先级并进行排序。

## 工作假设

以下内容仍需验证：

- 同类痛点是否广泛存在于其他 Codex 重度用户中。
- 用户最需要的是实时状态、异常提醒还是任务查找。
- 观察现有 Codex App 的状态推断准确度是否足以提供稳定价值。
- 用户是否接受把“缺少完成事件且长时间无活动”标记为“疑似异常”。
- 产品是否长期扩展到 Codex 之外的其他 Agent。

## 当前开放问题

1. 第一版只监控 `~/CodeX` 的长期项目，还是覆盖 Codex 中的全部对话？
2. 项目卡片最需要优先展示哪些信息？
3. 多久没有活动才应该提示“疑似异常”？
4. 用户希望第一版只显示状态，还是同时提供系统通知？
5. 临时目录、worktree 和长期项目应如何归并？
6. 在个人验证成功后，是否需要面向其他用户发布？

## 已做决策

- 使用独立长期项目空间，而不是只在聊天中记录。
- 产品讨论采用事实、假设、证据和决策分离的方式沉淀。
- 初步可行性判断为 `GO`，但这只是评估建议，仍等待用户决定是否进入原型阶段。
- 推荐第一版为本地只读看板，不替代 Codex App。
- 产品数据模型必须区分项目、任务 Thread 和单次执行 Turn。
- 不使用简单的“工作中 / 停止”二元状态；需要区分正在工作、等待处理、已完成、已中断、疑似异常、空闲和未知。
- 点击回到 Codex 对话可使用官方 `codex://threads/<thread-id>` 深链接。
- 现有 Codex App 的实时状态优先通过本地事件观察实现；在不能确定时必须显示“疑似异常”或“状态未知”。
- 产品形态提议为 macOS 本地桌面伴侣应用：菜单栏入口 + 桌面看板 + 后台只读观察器。
- 产品内部可以使用 Web UI，但第一版不做依赖云端的 Web 服务。
- localhost Web 页面只作为早期原型手段，正式交付应封装为可自动启动和菜单栏常驻的桌面应用。
- 产品形态决策记录在 `decisions/ADR-001-product-form.md`，已接受。
- 用户提出并认可度较高的内部架构方向：本地服务通过接口提供数据，页面使用 Web 方式呈现。
- 架构建议为“本地观察服务 + 版本化 REST API + SSE 实时事件 + Web 前端”，记录在 `decisions/ADR-002-local-service-web-ui.md`。
- API 只暴露归一化的产品对象，不暴露 Codex 原始数据库结构、对话正文或 JSONL 日志。
- 原型阶段由本地服务同源提供 Web 页面和 `/api/v1`，正式版本可把同一页面嵌入桌面壳。
- Local API v1 草案记录在 `docs/api-contract-v1.md`。
- 本地服务技术栈提议为 Go 1.26.x 单二进制，标准库优先，记录在 `decisions/ADR-003-go-local-service-stack.md`。
- Swift 可在后续承担 macOS 菜单栏或桌面壳，但不承担核心状态服务。
- Web 前端可以使用 TypeScript；第一版后台不使用 Node.js。
- 用户已确认 Go 技术栈并授权开始开发。
- ADR-003 已接受。
- 首个实现是只读命令 `agent-task-monitor probe`，先验证状态判断，不先做 HTTP 和页面。
- 第一版看板采用状态优先的本地任务指挥台：先显示工作中、需要关注、项目和任务总数，再显示项目卡片与关注动态。
- 页面使用原生 HTML、CSS 和 JavaScript，并通过 Go `embed` 打入单二进制；不使用外部字体、CDN 或前端运行时。
- ChatGPT Project 镜像只有在声明名称与 `~/CodeX` 中正式目录精确对应时才自动归并；证据不足继续待归类。
- 项目聚合状态、Codex 任务状态和后台进程状态必须分离；Codex 任务完成后，项目后台程序仍可使项目保持“工作中”。
- 后台进程观察只暴露运行时、脚本名、时间和推断状态，不暴露 PID、完整命令、参数或日志正文。
- 同一后台进程在 5 分钟内两次更换 PID 时，标记为“频繁重启／疑似异常”；决策记录在 `decisions/ADR-005-project-runtime-observation.md`。
- 看板桌面端采用三列紧凑卡片；全局状态分布条可点击筛选，卡片内用任务状态轨道和运行波形增强扫视效率。
- 没有异常且尚未选择项目时，关注与动态区域自动收起，避免重复内容阻挡项目列表。
- 项目网格强制使用统一弹性行高，卡片和底部操作栏保持对齐。
- 项目一句话介绍由 ProjectNavigator 统一生产；AgentTaskMonitor 按项目规范路径读取 v3 登记表，不在实时刷新中调用模型。
- 人工确认介绍优先于 README 等自动提取结果；资料不足时显示“待补充项目介绍”，不根据名称编造。
- 项目卡片用固定两行展示介绍，搜索同时覆盖项目名、介绍和任务标题；决策记录在 `decisions/ADR-006-project-summary-metadata.md`。
- 人工优先级独立保存在 ProjectNavigator 的 `project-preferences.json`，自动登记表和每日扫描不会覆盖。
- Codex 状态保持只读；本地服务仅能修改固定偏好文件，写入采用同源校验、输入限制、原子替换和上一版备份。
- 优先级为 P0–P3；同级支持拖拽和键盘上移／下移，异常关注区域不受人工优先级影响；决策记录在 `decisions/ADR-007-user-project-preferences.md`。
- 优先级选择或保存期间暂缓 SSE 和定时刷新；更早发出的刷新结果也会被丢弃，避免卡片重建打断原生选择器。
